# com.ragwatson

개발 노트·기획 문서(`docs/`)와 **에이전트 코딩용 하네스(harness)** 문서를 함께 두는 저장소다. 모델에게 긴 지시만 늘리기보다, 안드레 카파시가 에이전트 코딩에서 반복해 짚은 실패 패턴—**가정을 사용자 몰래 진행**, **혼란·모순을 표면화하지 않음**, **과도한 추상과 부수적 diff**, **말뿐인 목표**—을 **구조(규칙·검증·도구·사람의 검토)**로 줄이는 것이 이 README가 말하는 하네스 엔지니어링이다.

---

## 하네스 엔지니어링이란

**하네스**는 LLM 단독이 아니라, 그 주변에 얹는 안전망이다.

| 층 | 하는 일 |
|----|---------|
| **행동** | 구현 전 사고, 최소 코드, 정밀한 수정, 목표 중심 실행 → `CLAUDE.md` |
| **맥락·도구·검증** | 이 저장소에서 무엇을 믿고 어떻게 끝낼지 → `CURSOR.md` |
| **자동 검증** | 테스트·린트·타입·CI — “통과”가 에이전트 루프의 닻이 되게 한다. |
| **도구(MCP 등)** | 브라우저·API·DB 등은 가능하면 도구로 고정해 말로만 우회하지 않는다. |
| **사람** | 중요한 코드는 IDE에서 diff를 **요청과의 대응** 기준으로 본다. 에이전트만 믿고 멀어지지 않는다. |

카파시가 강조한 방향—**구체적 성공 기준을 주면 루프를 잘 도는 경향**, 지시만 늘리기보다 **검증 고리**를 쓰는 것—에 맞춘다.

**트레이드오프:** 하네스는 신뢰할 코드와 워크플로에 우선한다. 사소한 수정에는 판단으로 가볍게 적용한다.

---

## 이 저장소의 문서

- **`CLAUDE.md`** — 에이전트 **행동** 지침(Think Before Coding, 단순성, 정밀 수정, 목표 중심 실행). 프로젝트별 규칙과 **병합**해 쓴다.
- **`CURSOR.md`** — Cursor·프로젝트 **하네스** 설명: 무엇을 막고, 무엇을 두면 좋은지.

작업을 줄 때는 가능한 한 **검증 가능한 목표**로 바꾼다. 예: “버그 수정” → “재현 테스트 작성 후 통과”.

---

## 하네스가 잘 잡혔다는 신호

- diff에 **불필요한 변경**이 줄었다.
- **과도한 복잡도**로 인한 재작성이 줄었다.
- 구현 **전**에 모호함이 정리되는 대화가 늘었다.
- CI·테스트가 에이전트 작업의 **기본 종착점**이 되었다.

---

## Cursor에서 쓰는 법

Cursor가 루트 문서를 자동으로 넣지 않는 설정이라면, 사용자 규칙이나 `.cursor/rules`에서 `CLAUDE.md` / `CURSOR.md`를 참조하도록 연결하면 된다. (버전·설정에 따라 다름.)
