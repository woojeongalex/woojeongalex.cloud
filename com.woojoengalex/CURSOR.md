# CURSOR.md

Cursor에서 에이전트가 세션 시작 시 읽는 **프로젝트 하네스(harness)** 문서다. 모델 능력만 믿지 않고, 실패 패턴을 **구조**로 막는 것이 하네스 엔지니어링이다. 안드레 카파시가 에이전트 코딩에서 반복해 지적한 관찰—잘못된 가정을 사용자 몰래 진행, 혼란·모순을 드러내지 않음, 과도한 추상·부수적 diff, 말로만 된 지시—에 맞춰 이 파일을 구성한다.

**트레이드오프:** 하네스는 신뢰할 코드와 워크플로에 우선한다. 사소한 한 줄 수정에는 판단으로 가볍게 적용한다.

`CLAUDE.md`(행동 지침)와 **병합**해 사용한다. 행동은 `CLAUDE.md`, 이 저장소의 맥락·도구·검증 방식은 `CURSOR.md`에 둔다.

**기술 코딩 규칙:** `.cursorrules` §0 → `docs/DevOPs/README.md` → `docs/DevOPs/Backend/BACKEND_RULES.md` (코드 작성 전 필수).

---

## 1. 하네스가 막는 것 (의도 정렬)

에이전트는 여전히 **미묘한 개념 오류**를 낼 수 있다. 하네스는 다음을 전제로 짠다.

- 사용자 대신 **가정을 밀어붙이지** 않게: 모호하면 질문·대안 제시가 기본 행동이어야 한다 (`CLAUDE.md` 1절과 동일 선상).
- **검증 없는 확장 구현**을 줄인다: “작동하게”가 아니라 측정 가능한 성공 기준과 루프 (`CLAUDE.md` 4절).
- **범위 밖 리팩터·포맷·주석 손대기**를 줄인다: diff는 요청과 직결 (`CLAUDE.md` 3절).

---

## 2. 하네스 구성 요소 (무엇을 두면 좋은가)

| 구성 요소 | 역할 |
|-----------|------|
| **규칙 문서** (`CURSOR.md`, 필요 시 `.cursor/rules`) | 불변 제약, 용어, 디렉터리 규칙, 금지 사항을 짧고 명확히. |
| **행동 지침** (`CLAUDE.md`) | Think before code / 단순성 / 정밀 수정 / 목표 중심 실행. |
| **자동 검증** | 테스트·린트·타입체크·CI. “통과”가 에이전트 루프의 닻이 된다. |
| **도구(MCP 등)** | 브라우저·API·DB 등은 말로 우회하지 말고, 가능하면 도구로 고정해 환각을 줄인다. |
| **성공 기준을 선언형으로** | “이걸 해라”만이 아니라 “이 조건이 만족될 때까지 반복”이 드러나게 쓴다. |

원칙: **지시를 늘리기보다 검증 고리를 늘린다.** 카파시가 말한 “구체적 목표를 주면 루프를 잘 도는 경향”에 맞춘다.

---

## 3. 사람의 자리 (하네스 밖이 아님)

- **중요한 코드는 IDE에서 같이 본다.** 에이전트만 믿고 멀어지지 않는다.
- diff는 **요청과의 대응**을 기준으로 본다. 변경 이유가 한 줄이라도 불명확하면 되돌리거나 질문한다.

---

## 4. 이 저장소에서의 적용

- 세션마다 `CLAUDE.md`와 본 `CURSOR.md`를 함께 전제로 둔다.
- 작업 제안 시 **검증 단계**(예: 어떤 테스트/명령이 녹색인지)를 같이 적는다.
- 프로젝트에 테스트·스크립트가 있으면 그것을 성공 기준의 기본으로 삼는다.

---

**하네스가 잘 잡혔다는 신호:** 불필요한 diff 감소, 재작성 감소, 구현 전에 모호함이 정리되는 빈도 증가, CI·테스트가 에이전트 작업의 기본 종착점이 됨.
